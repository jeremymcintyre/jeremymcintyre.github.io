Code-sharing favorites — who wins? Inheritance vs. Composition

16 Nov 2014

You are designing an application. You opt for one of two scenarios. In the first, code is repeated and logic duplicated when and where it's needed. There is some reuse of code, but it's random. In the second, each piece of knowledge is written only once, and then implemented from its source.

You choose the second.

Why do you choose the second? Because it is smarter. When you need to change, update, or extend your code in the future, this means you only have to make changes to each relevant piece of logic in one spot, rather than hunting it down and changing it all over the place — fewer bugs, less maintenance. It also makes your code more modular ("Cool! I can reuse that over here...") and more orthogonal ("When I change something over here, nothing else breaks.") This concept is essentially what Andy Hunt and Dave Thomas popularized under the D.R.Y. acronym — "Don't Repeat Yourself."

Another complementary idea is the Single Responsibility Principle (SRP), which suggests that each class or method should have a single responsibility. This promotes modularity further, since we aren't faced with dissecting simple behaviors out of more complex chunks of behavior when we want to reuse them.

Last before we get to the heart of our subject matter is the Law of Demeter (LoD). It came out of Northeastern University (my alma mater!) back in the late '80s, and basically promotes loose coupling. I like to think of it as the "Law of Making Your Code Run Like An Intelligence Agency," since it holds that units of code should know as little as possible about other units of code, communicate on a need-to-know basis only, and not talk to units of code that they have no business talking to. This prevents bugs and other problems from arising by minimizing illogical or promiscuous dependencies (entanglements) in your code.

Great! We are now ready to start introducing our subject matter. Please go ahead and keep this stuff in the back of your mind — we'll tie it in as we go.

Inheritance and composition are two Object-Oriented Programming (OOP) concepts that support the reuse of code. Inheritance creates relationships described in terms of "is-a," as in Ocelot "is-a" Cat. Composition creates relationships articulated in terms of "has-a," which are much more flexible than ones forged through inheritance. However, since inheritance is the easier one to explain, we'll start there.


class Cat
  attr_reader :what_am_i

  def initialize
    @what_am_i = "I'm a cat."
  end

end

class Ocelot < Cat
end

The benefit of inheritance is that it decreases redundancy in code by allowing child classes to inherit common general, abstract behaviors from a parent class. We may never want to directly create an instance of Cat, so it remains an abstract holding area for behaviors that would be useful for any class that "is-a" Cat to inherit. Since Ocelot "is-a" Cat, it inherits access to all the code within Cat.


tufted_ears = Ocelot.new
tufted_ears.what_am_i

=> "I'm a cat."

Another useful benefit of inheritance is that children classes can override methods from the parent class, or change them to make them more specialized. Take a look again below:


class Cat
  def what_am_i?
    "I'm a cat."
  end
end

class Ocelot < Cat
  def what_am_i?
    super + "..or more specifically, an ocelot."
  end
end

generic = Cat.new
tufted_ears = Ocelot.new

generic.what_am_i?
=> "I'm a cat."

tufted_ears.what_am_i?
=> "I'm a cat...or more specifically, an ocelot."

Inheritance is D.R.Y. since it cuts down on redundant code and helps keep classes modular...and I suppose it can be pretty effective in conjunction with Single Responsibility Principle...but what about the Law of Making Your Code Run Like An Intelligence Agency (LoD)? So long as the "is-a" relationship remains true, inheritance is fine; the need-to-know basis is real because the child classes are specialized extensions of the parent class. However, the minute this starts to change, red flags should go up. Many smarter and more experienced voices will tell you that as projects develop, classes that started out seemingly sharing an "is-a" relationship eventually diverge enough where that sort of relationship doesn't really make sense any more...and becomes an ungainly burden. It's situations like those that make them prefer composition over inheritance.

Whereas inheritance involves giving one class complete and immediate knowledge of another class, composition does not. Instead, it uses classes or modules directly in order to accomplish what it needs. It is best served with an example, so let's use composition such that Ocelot "has-a" Cat.


class Cat
  def identity
    "Cat"
  end

  def purr
    "purrr"
  end
end

class Ocelot
  def initialize
    @cat = Cat.new
  end

  def i_have_a
    @cat.identity
  end

  def make_sound
    @cat.purr
  end
end

little_spots = Ocelot.new
little_spots.i_have_a
=> "Cat"

little_spots.make_sound
=> "purrr"

As you can see, Ocelot initializes an instance of Cat and uses it directly by storing it in an instance variable. This way, the class does not have immediate knowledge or access to all methods of Cat...only the ones that we call on the instance variable, and then only from within methods of Ocelot. So little_spots.identity will throw an error because Ocelot did not inherit from Cat, it just uses an instance of it. See how this is much more of a need-to-know basis? It decouples the two classes much more than inheritance does, while also providing sharing of specific, relevant behaviors. This flexibility is what makes composition an oft-preferred alternative to inheritance.

There we have it. I won't go into it here, but using Modules is also composition. And, while inheritance is cool, initially convenient, and sometimes totally appropriate, I think the answer to our title is that composition is my code-sharing favorite.

As always, beware of sweeping declarations and the like — and be sure to consider inheritance and composition for yourself. I will continue to do the same.

Hope this was helpful, and thanks for reading!
