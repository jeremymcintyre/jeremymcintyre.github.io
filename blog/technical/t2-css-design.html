What are the best practices for using classes and ids?

19 Oct 2014

Imagine looking at a magazine article. In your mind's eye, consider all the components of the article - there are titles, subtitles, paragraphs of text, perhaps isolated boxes of text highlighting some key concepts regarding the article. That's like the HTML of a website - it's the structure; the "what" as in "what's on the page?"

Now consider all the little design decisions that needed to be made to get the magazine article looking so nice and professional - the margins, the colors, the sizes of different kinds of text, the placement of each structural component. That's like the CSS - it targets different objects in the HTML (for our sake, the magazine article) and tells them where to go and what to look like.

Let's drive this point home: if all you had were the unformatted components of the magazine article, it would be like having just the HTML of a webpage.

Now, how would you tell the different pieces where to go and what to look like? Well, I suppose you would call them out by name. "Hey, Title, go near the top of the page - a little left...perfect." This would be the equivalent of using the HTML <title> tag as a selector, so that your CSS would know where to apply your directions.

In other words, you use the tags built into the HTML to select the content and change how it displays with CSS. In HTML, tags generally bookend content such as this: <title>This is the title</title>. As you can see, there is an opening and a closing tag, and the content is in between. So if you wanted to make the title bigger, you would select it with the title tag, and then apply CSS to the tag.

The same would be true for paragraphs, so if you wanted to format paragraphs a certain way, you could select the HTML tag for paragraphs, <p>, and then apply your CSS to tell them how to look.

However, let's say you have a standalone box on the magazine page that is formatted differently than the rest of the article. It has a paragraph in it, but you want it to look different than the other paragraphs. This would be a good case to use an ID, because there is only one of those standalone boxes, and you will only need to use the styling rules in that one case.

We'll get back to IDs in a second, but first let's introduce classes as well. While IDs are perfect when you only have one instance of something, classes are great when you want to reuse similar styling for other components. For example, if you consider not just the magazine article but the entire magazine, chances are you'd have some styling rules that you'd want to reuse. Maybe you would want the main body paragraphs of all articles to look more or less the same, so you could create a set of CSS rules for that class. Then, every time you wanted those rules to be applied to a new article, you could just add the class name to the appropriate content and it would be applied!

At this point, I think we can mostly shed our magazine metaphor, though we might keep it handy in the future. However, let's dive into some of the web-based advantages and good practices of using classes versus IDs, now that we have an idea of what they are and why they might be used.

(Just in case you're still sorting it out, let me summarize up this point: classes and IDs are ways to specify and select specific content from an HTML structure.)

If classes and IDs both select content in an HTML document, and there are also existing generic HTML tags like '<title>' and '<p>' that can select content in an HTML document, how are they all different? The answer can be summed up mostly in one word: specificity.

Generic HTML tags are really generic. Classes are made by you, so they can be more specific. IDs are the most specific, because each one can only refer to one place in the HTML document. Therefore, there are particular uses that are better suited to classes, and some better suited to IDs.

Not every HTML element needs to be assigned a class or an ID. On the other hand, it is not exclusive that an item with a class or multiple classes cannot also have an ID; they can have both. But as we said, if there is a particular element that will have unique styling, then an ID is appropriate. Further, because IDs can be used in links, they can be more appropriate for titles, headers, or other places that a user might link to from another place. However, if a set of CSS rules might be applicable in more than one place, then it is best to assign a class instead.

So now we know that elements can have more than one class, and they can have an ID in combination with a class or classes, and finally that we can utilize these selectors to create very specific rules in our CSS. Classes can help these rules be applied in distinct cases, while IDs are perfect for unique styling situations, or to mark an element as a destination for a hyperlink. We also know a bit about how Not to use classes and IDs, and can think about when it might be appropriate to use each.
