The "what" and "how" of the Internet

30 Nov 2014

Most people are comfortable with their understanding of what the internet is, and are content to know that it works without knowing precisely how or why it works.

In stopping to think about it, one could suppose that it is a network of interconnected networks. This much seems obvious from its name.

But perhaps it is less intuitive to think of the internet as a system of hardware and protocols. Yet that is precisely what the internet is made of: hardware, and protocols that tell hardware how to handle information.

Many sources seem to explain the internet starting with your personal computer and building outward to the higher level connections of hardware and protocols. In the interest of a fresh perspective, I am going to do the exact opposite.

Internet Hardware and Infrastructure

The highest level of our heirarchy is the Network Service Provider (NSP). Each NSP is like the king or queen of its kingdown, and can handle all of the traffic from devices connected to its networks. Their hardware forms a high level internet backbone. If two computers are connected to networks under the same NSP, their communications would at most have to travel up to the level of NSP hardware (i.e. cables) before being routed back down the heirarchy to the appropriate destination computer.

However, because every device connected to the internet can theoretically communicate with any other device connected to the internet, these massive NSP corporations must have the ability to share traffic so that a computer under one NSP can communicate to a computer connected to any other NSP's networks. In order for this to be possible, there are agreements for the free transfer of information between NSPs at places called Internet Exchange points (IXs) of the two varieties: Network Access Points (NAPs), and their privately-owned equivalents, Metropolitan Area Exchanges (MAEs).

So, we know that Network Service Providers share information traffic to make global communication possible. Now let's focus on the heirarchy under a single NSP.

Network Access Points (NAPs), Metropolitan Area Exchanges (MAEs)

Below each Network Service Provider there are a number of smaller Internet Service Providers (ISPs), which manage smaller networks. For example, an NSP might have regional ISPs under it, and local ISPs under those. ISPs either have their own networks and internet hardware backbones, or rent bandwidth from their higher-ups on the heirarchy.

Internet Service Providers possess banks of hardware to serve access to the network, and in the end connect to our personal devices through one means or another. ISPs typically provide internet through Wide Area Networks (WANs) which are then connected to homes, office buildings, and more. One of these smaller networks is called a Local Area Network (LAN).

Ultimately our personal devices connect to a router that manages the traffic in the Local Area Network. The hardware that connects to the internet such as laptops, smartphones, and tablets make up the bottom of the internet hardware heirarchy.

The Roles of Protocols

So when information is shared on the internet, how does it know where to go? In order to make this possible, hardware follows certain protocols that ensure information is delivered to the appropriate destination.

Each time you send information from an application on your computer to another computer on the internet, the information passes down four protocol layers, over the internet, and then back up the four protocol layers on the receiving machine. These layers address how to display the data, what app it should be delivered to, what computer it should be delivered to, and the conversion between binary data and network signals. Here is a look at those layers:

Layer Description
Application Protocol Protocols specific to a given application (email, www, FTP, etc.)
Transmission Control Protocol

Directs data to a specific application via port number.

"What app should this go to?"
Internet Protocol

Directs data to the appropriate IP address on the internet.

"What computer should this go to?"
Hardware Layer Converts binary data to network signals and back again.

As an example, when you send an email it starts at the top of the stack. As it travels down the stack, it may be broken up into smaller chunks called data packets. The applications protocol layer knows that these packets are an email, and the TCP layer adds a header to each packet with an application-specific port number. This is done so the receiving computer knows what application to send it to. The next step in the process is to tag each packet so that it is sent over the internet to the correct computer.

Each piece of hardware must have its own unique address for this to work. This address is called an IP address (for Internet Protocol). Devices permanently connected to a LAN might have a permanent IP address, but it is not important that IP addresses be permanent; only that they are unique to a specific device while it is connected to the internet.

NOTE: an IP address is of the form n.n.n.n, where each n is a number between 0 and 255. This range arises from the fact that each n is expressed as an octet of binary numbers (either 0s or 1s). It is only possible to express 28 (256) unique combinations in a binary octet, so each combination represents a number in that range.

Now, here is where routers (more hardware) become extremely important. After your data packets are translated into network signals, they are sent to your router.

Routers typically exist as buffers and information managers between networks. For example, your home router knows the IP addresses of all devices connected to it, and probably bridges between your Local Area Network and the ISP's Wide Area Network. When it receives packets of data, it determines whether it knows the destination IP address tagged on each packet. If it does, it sends it to the correct device in your network. If not, it follows a default action — usually to send the packets up the infrastructure heirarchy. The next router does the same, sending it upward again if necessary. In this way, routers direct packets of information across the internet.

But I don't know any IP addresses...

Good point. How does it get worked out what IP address we want to interact with?

The answer is the Domain Name System (DNS), which maps IP addresses with their domain names so we don't have to consider IP addresses when using the internet. For example, google.com is a domain with the top-level domain of .COM.

When navigating the internet, our browsers send requests to DNS servers to get the IP addresses that match up with the names we provide in the Uniform Record Locator (URL). If the DNS server doesn't know the IP address, it passes it around to other DNS servers, or else returns an error that the domain name doesn't exist.

Summary

The internet is a network of networks made up of hardware and protocols that emsure information is delivered where it is intended to go. Routers, servers, and other types of hardware assist in all of this. Obviously we could go into much more detail about how certain protocols work like HTTP/HTTPS, FTP, and even how caching works - but this is just intended to be a short introduction to how the internet works and what it is. I hope it was clear and interesting, and otherwise, thanks for reading!
