Ruby hashes and JavaScript objects: how similar are they?

23 Nov 2014

This week I began learning JavaScript, and I must say I like it a lot. It feels very different from Ruby, but also familiar. There are plenty of functional and syntactical contrasts to be made, but also elements that even look somewhat similar at first glance. One example of this is in the comparison between Ruby hashes and JavaScript objects.

In a lot of ways, they are pretty similar. Look at them!

  # Ruby

  my_hash = {penguins: 42, porpoises: 12}
  => {:penguins => 42, :porpoises => 12}
        
  // JavaScript

  myObject = {penguins: 42, porpoises: 12};
  > {penguins: 42, porpoises: 12}

Both of them are dictionary-like structures with associated pairs, the first of each pointing to the second; both are also objects (following Ruby's almost-true "everything-is-an-object" model); and, both allow their values to be changed. However, there are also a number of differing features between the two.

A good place to start might be "how to talk about them." In Ruby hashes, the first part of each pair is called a key, which would be similar to a property in a JavaScript object.


  {key1 => value1, key2 => value2}
        

  {property1: somethingHere, property2: somethingElse}

Keys point to "values" in hashes, which can be anything from Boolean values to Fixnums, Floats, Strings, Arrays or other Hashes. A JavaScipt object's properties can similarly store numbers, arrays, Boolean values, other objects, or even functions. Storing methods in hashes is unsuccessful, because Ruby tries to evaluate the method and then store the result, whereas properties can store functions and use them in a way syntactically reminiscent of calling a method on a Ruby class.


  def hello_from_hash
    "Hi"
  end

  hash = {my_method: hello_from_hash}
  => {my_method: "Hi"}
